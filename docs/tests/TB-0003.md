## How to run (manual, HTTP-level)

1. `docker compose -f compose.e2e.yaml up -d --wait`
2. Sign up a customer: `POST /api/signup` with a check-digit-valid CPF (bureau default-approves).
3. Obtain a bearer token via the real PKCE flow (browser, or the `PkceTokenFlow` test helper if
   running from inside the backend test suite).
4. Fund the account directly: `INSERT INTO posting (...) ... ; UPDATE balance SET amount = ...`
   against the `db` service (resolve its container via `docker compose -f compose.e2e.yaml ps -q
   db`, not a hardcoded name).
5. Exercise the routes below with `curl --max-time 15`.

## Findings

### QA-0003-01 — LOW/MEDIUM — fixed post-review by the main agent — type-coercion failures on `size`/`from`/`to` leak Spring/Java internals in the 422 body

Every hand-rolled validation on this endpoint (`direction`, `size` range, `cursor` shape, `from`
after `to`) returns a clean domain-language `detail`. A value that fails **type coercion** before
any hand-written validation runs (`size=notanumber`, `from=2026-07-01` without a time component)
instead echoes Spring's own conversion-failure message verbatim, naming Java types
(`java.lang.Integer`, `java.time.Instant`) and the exact `@RequestParam` annotation. Root cause
(confirmed by reading `StatementController`/`ApiErrorHandler` and checking `MethodArgumentType-
MismatchException`'s real class hierarchy against the actual `spring-web` jar on the classpath):
the pre-existing, unchanged `ApiErrorHandler.unacceptableSubmission` handler is reached via
Spring's cause-chain-based `@ExceptionHandler` resolution (the type-mismatch exception's
underlying cause is an `IllegalArgumentException` subtype), but the handler echoes the OUTER
exception's own generic message, not a domain one. Still `422`, never `500`, no stack trace, no
secret/PII disclosed — a hardening/consistency gap, not a security or data defect.

**Fix applied**: `from`/`to`/`size` are now bound as raw `String` `@RequestParam`s and parsed
explicitly (`Instant.parse`/`Integer.parseInt`), exactly like `direction`/`cursor` already were —
never a typed `@RequestParam` that Spring converts automatically. A type-coercion failure now
raises the same curated `IllegalArgumentException` every other validation on this endpoint uses,
producing a domain-language `detail` (`"size must be a whole number, was notanumber"`, `"from must
be a full ISO-8601 instant, was 2026-07-01"`) instead of a Spring-internal one. Regression test:
`StatementControllerIT.refusesUnparsableFiltersWithoutLeakingFrameworkTypes`.

### QA-0003-02 — MEDIUM — fixed post-review by the main agent — the statement screen (and its own new manual section) promise cross-month navigation that does not exist

The screen offers a direction filter and "Load more" pagination, but never sends `from`/`to` — so
"Load more" only ever paginates within the current month, permanently. Verified live: a customer
with a posting one minute before the current month's start never sees it, in any number of
"Load more" clicks; a direct HTTP call with an explicit period **does** return it, proving this is
a frontend gap, not a backend limitation. This delivery's own new manual text ("choose **Load
more** to keep going further back") overstates what the shipped screen does. No acceptance
criterion is violated (none names a period-selector control), and no money-safety/authorization
property is affected.

**Fix applied**: the statement screen now has a period navigator (`‹`/`›` buttons beside a month
label) that explicitly requests the calendar month being viewed, computed client-side at the
product's fixed UTC-3 timezone to match `StatementFilter.currentMonth`'s own boundary exactly;
"Load more" continues to paginate within whichever month is selected. The manual's wording was
corrected to describe this actual mechanism (move between months via the arrows; "Load more" for
older lines within the selected month) rather than implying "Load more" alone reaches earlier
months. Regression test: `statement.spec.ts`'s "moves to the previous month and back" case.
