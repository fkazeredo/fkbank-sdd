# TB-0001 — Test book, ledger core (SPEC-0001)

Risk R3. Branch `feature/spec-0001-ledger-core`, head `f7bf0f9`. Verdict recorded in
`docs/qa/QA-0001.md`.

Environment for every case below: real PostgreSQL 16 via Testcontainers (acceptance suite) or a
throwaway PostgreSQL 16 container with `V1`/`V2` applied (direct-SQL cases). No in-memory profile
is used anywhere — acceptance criterion 4 forbids it and the rest of the criteria are only
meaningful against the engine that actually runs.

## How to run

```bash
cd backend
./mvnw -o verify \
  -Dit.test='LedgerAcceptanceIT,LedgerRaceAcceptanceIT,LedgerMetricAcceptanceIT' \
  -Dtest='LedgerAcceptanceIT' -DfailIfNoTests=false \
  -Dsurefire.failIfNoSpecifiedTests=false -Djacoco.skip=true
```

Expected at the time of writing: `LedgerAcceptanceIT` 20/20 green; `LedgerRaceAcceptanceIT` 22
run with 10 failures, all of them the two error-contract cases that carry finding QA-01;
`LedgerMetricAcceptanceIT` 2 run with 1 failure carrying QA-03. Those failures are the findings,
not flakiness — each reproduced identically on every repetition.

## Lock mutation

Performed on an isolated copy of the backend outside the repository, because the path-guard hook
correctly forbids QA from editing production code. With `@Lock(PESSIMISTIC_WRITE)` removed from
`BalanceJpaRepository`, the builder's `LedgerConcurrencyIT` fails with `[COMMITTED, COMMITTED]`
and a lost update, and the QA race suite fails with `expected: 1L but was: 2L` and balances of
`BRL 60.0000` / `BRL 125.0000` against a correct `BRL 85.0000`. With the lock in place both are
green. The race tests prove the lock rather than coexist with it.

## Known test-suite hazard

`LedgerSchemaIT` inserts postings directly into the table without moving the matching balances.
Postings are append-only, so that drift can never be cleared for the rest of the JVM's life. Any
later assertion that `driftedAccounts()` is globally empty — `LedgerVerificationIT`
`.flagsNothingOnACleanLedger` does exactly that — passes only because failsafe currently runs
`com.fkbank.infra.*` after `com.fkbank.domain.*`. Recorded as QA-04. The QA suite in this book is
deliberately immune: it asserts drift only for the accounts each test opened.

**Closed in run 2.** `insertPosting` now moves both settlement balances to match every posting it
writes, and the failing inserts throw before the balance statements run. Verified by executing the
battery with the class order reversed rather than by reading the code: `Ledger schema` ran 3rd and
`Ledger - trial balance` 12th, and all 89 integration tests passed.

## Mutation: do the new regression tests have power

Both fixes were reverted on a copy of `backend/` outside the repository, keeping the post-fix
tests. The repository was never modified.

| Test | Owner | Against pre-fix code |
|---|---|---|
| `LedgerConcurrencyIT.concurrentReversalsProduceOneContraAndAStableRefusal` | Builder | **FAILS** — genuine |
| `LedgerSchemaIT.refusesToTruncateThePostingTable` | Builder | **FAILS** — genuine |
| `LedgerMetricsIT` (all three) | Builder | **PASSES 3/3 — vacuous, QA-09** |
| `LedgerMetricAcceptanceIT.aRolledBackPostingIsNotCounted` | QA | **FAILS** — genuine |

`LedgerMetricsIT.ignoresARolledBackPosting` rolls back via `InsufficientFundsException`, a path on
which `record` throws before `events.publish` is ever reached — so no event is published under
either listener and the assertion cannot fail. The behaviour it names is in fact guarded only by
the QA-owned acceptance test above.
