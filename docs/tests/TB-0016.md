## Findings

See `.claude/runtime/SPEC-0016/qa-report.md` Â§Findings for full detail, repro commands, and root
cause.

1. **[BLOCKER]** `/actuator/prometheus`, `/actuator/health`, `/v3/api-docs`, and
   `/swagger-ui/index.html` are unreachable through the deployed single-origin edge
   (`http://127.0.0.1:8090`) â€” `frontend/nginx.conf` has no proxy rule for these path prefixes,
   so they silently fall through to the Angular SPA shell instead of reaching the backend. The
   backend's own implementation is correct (verified directly against the container, bypassing
   nginx). Executable repro: `frontend/e2e/observability-edge.spec.ts` (QA-owned, all 5 cases
   red against the current branch, each failing for the documented reason).
2. **[Finding, recorded for owner visibility, not independently blocking]** No log line carrying
   `correlationId` is produced by the deployed backend for any ordinary request, in any profile
   â€” only a `DEBUG`-level line exists and no profile enables `DEBUG` for that logger. The MDC
   propagation mechanism itself is sound and future-proof (any later business logging will
   automatically carry the id), but today's deployed artifact provides zero log-based per-request
   traceability. Recommend the owner decide whether to add a minimal `INFO`-level access-log line
   as a Day-0 complement, or explicitly accept this as a skeleton-phase trade-off (parallel to the
   OR-2 "no subject yet" sequencing note already in the spec).
3. Minor observations (not blocking): `/v3/api-docs.yaml` inconsistently `401` while `/v3/api-docs`
   is public; `/swagger-ui/` (bare path) `401` while the spec-named `/swagger-ui/index.html` works.
