# TB-0002 — Test book, sign-up and account opening (SPEC-0002)

Risk R3. Branch `feature/spec-0002-signup-account`. Verdict: `FAIL_REWORK` (run 1 of 2).

Environment for every HTTP-level case below: the whole application on a real port
(`@SpringBootTest(webEnvironment = RANDOM_PORT)`) against real PostgreSQL 16 via Testcontainers,
`e2e` profile, the bureau replaced by a controllable double (`ControllableBureau`) for
determinism — the real bureau container, the real network call and the real client-side timeout
are exercised separately, by the browser journeys below and by the emulator's own contract
tests. No in-memory profile is used anywhere.

## Findings

### QA2-01 — MEDIUM — the bureau callback does not bind `inquiryId` to the onboarding it decides

`BureauCallbackController.receive` parses `inquiryId` off the signed body
(`BureauCallbackController.java:92-93`) and never reads it again —
`grep -rn inquiryId backend/src/main` finds it nowhere else in production code, and
`OnboardingOutcome.apply(OnboardingId id, BureauDecision decision)`
(`OnboardingOutcome.java:62`) has no inquiry parameter to check it against.
`BureauClient.decide` (`BureauClient.java:54-68`) parses the bureau's own `inquiryId` off the
*synchronous* response and discards it the same way. The onboarding never records which inquiry
it was actually opened under.

**Reproduced**: a validly HMAC-signed callback carrying a completely fabricated `inquiryId`, but
the real onboarding id (public — returned to the applicant in the `202` body), is accepted and
applies its decision. Manually, against the real emulator: forging a `REJECTED` decision for a
still-`PENDING` onboarding permanently overrode the real `APPROVED` decision the bureau sent a
moment later — the genuine, correctly-`inquiryId`d callback landed afterward and was silently
treated as a no-op because the application was already "decided."

```
$ ./mvnw -o verify -Dit.test='BureauCallbackAcceptanceIT#aFabricatedInquiryIdShouldNotBeAbleToDecideSomeoneElsesApplication' \
    -Dtest='none' -DfailIfNoTests=false -Dsurefire.failIfNoSpecifiedTests=false -Djacoco.skip=true
Tests run: 1, Failures: 1
  expected: not equal to 200 but was 200 (and the onboarding is no longer pending)
```

Given a compromised or leaked HMAC secret — or a bureau that reissues a fresh inquiry for a
retried check — nothing in the design distinguishes a genuine completion from a fabricated one
beyond the shared secret and a known, still-open onboarding id. Direction only, not an edit:
persist the `inquiryId` the synchronous call (or the pending state) was opened under, and refuse
a callback whose `inquiryId` does not match it.

### QA2-02 — LOW — a concurrent loser's fallback can miss an already-decided winner

`SignUp.submit()`'s recovery path for a losing concurrent insert
(`SignUp.java:100-108`) calls `onboardings.findPendingByCpf(cpf)`, which is scoped to
`status = 'PENDING'` (`OnboardingJpaRepository.findByCpfAndStatus`,
`onboarding_one_pending_per_cpf`'s own migration comment: *"the loser reads the winner's row and
returns it"* — with no qualification on the winner's status by the time that read happens). Once
the winner has been decided — which the synchronous `approve`/`decline` bureau path can complete
before a genuinely raced loser gets to retry — that lookup returns nothing, and `SignUp.submit()`
re-throws `OnboardingAlreadyPendingException`, surfacing to the caller as
`409 ONBOARDING_ALREADY_PENDING` about an application that has, in fact, already been decided.

**Reproduced deterministically** (the mechanism, not the exact interleaving — see below):

```
$ ./mvnw -o verify -Dit.test='SignUpConcurrencyHttpAcceptanceIT#theFallbackLookupStillFindsADecidedWinner' \
    -Dtest='none' -DfailIfNoTests=false -Dsurefire.failIfNoSpecifiedTests=false -Djacoco.skip=true
Tests run: 1, Failures: 1
  expected: present but was: empty
```

**Black-box evidence of the real timing window**: reproduced once in 13 true-HTTP-concurrency
trials (a 5-way race under the `approve` scenario; the other 12 trials, 2-to-8-way, all produced
the graceful `200`/PENDING response). Row-count safety held even in the anomalous trial — exactly
one customer, one onboarding, one account. Severity is LOW because no data integrity is at stake
and the window is narrow; it is still a real deviation from BR-5's own wording and from the
schema's own comment for whichever caller is unlucky enough to hit it.

No automated test attempts to force the exact interleaving: doing so would require either a
genuinely flaky timing-dependent test or injecting an artificial delay into production code,
which QA does not edit. The deterministic test above proves the mechanism the flaky HTTP evidence
is explained by, without pretending to reproduce the race itself on demand.

### QA2-03 — MEDIUM — the OpenAPI snapshot documents only `200` for every onboarding/account route

`GET /v3/api-docs` documents a single `200` response for `POST /api/signup`,
`GET /api/signup/{onboardingId}`, `POST /api/webhooks/bureau` and `GET /api/account/me`. None of
`201`, `202`, `401`, `404`, `409` or `422` is documented anywhere, despite every one of them being
real, deliberate, and BR-1/BR-3/AC-2-required — confirmed both against the real running server and
by `grep -rn ApiResponse backend/src/main/java/com/fkbank/application/api` returning nothing (no
response annotations exist to describe them). `OpenApiSnapshotIT` (builder-owned) only proves the
served document has not drifted from the committed snapshot — it cannot catch this, because the
snapshot is generated from the same unannotated controllers this finding is about.

```
$ ./mvnw -o verify -Dit.test='OnboardingContractAcceptanceIT' -Dtest='none' -DfailIfNoTests=false \
    -Dsurefire.failIfNoSpecifiedTests=false -Djacoco.skip=true
Tests run: 4, Failures: 4
```
