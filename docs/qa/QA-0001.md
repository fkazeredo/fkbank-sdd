### Black-box findings (frozen)

**BB-1 ‚Äî MEDIUM ‚Äî A concurrent repeat reversal never returns `REVERSAL_NOT_ALLOWED`.**

BR-6 and acceptance criterion 7 promise the stable code `REVERSAL_NOT_ALLOWED` for a repeat
reversal, and `docs/DOMAIN.md` ¬ßVerifiable invariants states the same. Sequentially that holds.
Under concurrency the loser of the race gets a *different* error, in two variants, both
reproduced 5/5 ‚Äî deterministic, not flaky:

- *Reversed amount no longer available* ‚Üí `InsufficientFundsException`:
  `account 69 holds BRL 0.0000 and cannot cover BRL 30.0000`
- *Funds available* ‚Üí the raw persistence error escapes the ledger API:
  `org.springframework.dao.DataIntegrityViolationException ... constraint
  [posting_reverses_at_most_once_idx] <- org.hibernate.exception.ConstraintViolationException
  <- org.postgresql.util.PSQLException`

The money is safe in both cases ‚Äî exactly one contra exists, balances are correct and
`trialBalance` stays consistent. What breaks is the published error contract every future money
rail will consume: a duplicate refund would surface to the customer as "insufficient funds", or
as an unmapped infrastructure exception (an HTTP 500 rather than a clean business error). It
also means a Spring `dao` type crosses out of a `domain` API call.

Repro (both are QA-owned, already committed and currently failing by design):

```

$ ./mvnw -o verify -Dit.test='LedgerRaceAcceptanceIT#theLoserOfAFundedReversalRaceGetsTheStableDomainError' ...
Tests run: 5, Failures: 5
```

**BB-2 ‚Äî MEDIUM ‚Äî `TRUNCATE posting` bypasses the append-only guarantee and empties the ledger.**

M1 ("append-only ledger ... enforced by a DB trigger") and acceptance criterion 3 ("row #N stays
immutable, no UPDATE/DELETE") are enforced by a row-level `BEFORE DELETE OR UPDATE` trigger.
PostgreSQL row-level triggers do not fire on `TRUNCATE`, and the application's own database role
owns the table:

```
=== ATTACK 1: UPDATE a posting ===
ERROR:  posting is append-only: UPDATE on posting is not permitted
=== ATTACK 2: DELETE a posting ===
ERROR:  posting is append-only: DELETE on posting is not permitted
=== ATTACK 3: TRUNCATE posting ===
TRUNCATE TABLE
=== postings remaining ===
 postings
----------
        0
```

The trigger function is already `TRUNCATE`-aware ‚Äî it produced the correct message the moment a
statement-level trigger was attached ‚Äî so only the registration is missing. Verified remedy
(QA-executed on the probe container, offered to the builder as a direction, not as an edit):

```sql
CREATE TRIGGER posting_no_truncate BEFORE TRUNCATE ON posting
  FOR EACH STATEMENT EXECUTE FUNCTION posting_is_append_only();
```

```
=== PROPOSED FIX: statement-level TRUNCATE guard ===
CREATE TRIGGER
ERROR:  posting is append-only: TRUNCATE on posting is not permitted
=== postings after the guarded TRUNCATE attempt ===  postings = 2
```

**BB-3 ‚Äî LOW (observation) ‚Äî The schema permits a reversal chain; only Java forbids it.**

`posting_does_not_reverse_itself` blocks self-reference and
`posting_reverses_at_most_once_idx` blocks two contras for one original, but nothing stops a row
whose `reverses_posting_id` points at a posting that is itself a reversal:

```
=== ATTACK 6: reverse a REVERSAL at DB level (chain) ===
INSERT 0 1
 22222222-... |
 33333333-... | 22222222-...
 55555555-... | 33333333-...
```

BR-6 only requires the index to back the *at-most-once* half, so this is within spec. It carries
no race window either ‚Äî `isReversal` is a property of an immutable row, so the Java check cannot
go stale. Recorded because a later rail could reasonably assume the database enforces both halves.

**BB-4 ‚Äî LOW (observation) ‚Äî Nothing in the schema stops a negative customer balance, and the
application role can disable the guard.**

```
=== ATTACK 4b: drive a CUSTOMER balance negative by direct UPDATE ===
UPDATE 1
 account_id |        kind        |   amount
        100 | CUSTOMER_AVAILABLE | -50000.0000

=== ATTACK 7: disable the append-only trigger as the application user ===
ALTER TABLE
```

The `balance` table carries no `kind`, so a plain `CHECK` cannot express "customer accounts only";
the specified mechanism (BR-5 pessimistic locks + the domain guard) is what holds the line, and
`trialBalance` does detect the resulting drift. Both facts are ordinary consequences of the
application owning its schema. Recorded as defence-in-depth context for the security gate rather
than as a defect against this spec.

### Black-box conclusions

- Criteria 1‚Äì6 pass literally, in the environment each names.
- Criterion 7 passes sequentially and holds its *money* guarantee under concurrency, but not its
  *error contract* (BB-1).
- I could not make a customer or box account go negative by any application route: not by
  posting, not by reversing a spent deposit, not by racing two drains, not by racing a reversal
  against a spend. Direct SQL can, and the verification routine catches it.
- I could not get a posting reversed twice, nor mutate or delete one through the application or

- `trialBalance` genuinely reads the `posting` table: it flags drift even when the totals are

- `Money` did not round mid-calculation in any probe, kept 4 decimals through the PostgreSQL

## Pass 2 ‚Äî White box (opened only after the block above was written)

### Root cause of BB-1, confirmed in the code

`backend/src/main/java/com/fkbank/domain/ledger/Ledger.java:73-85` runs both reversal guards
*before* any lock exists:

```java
public Posting reverse(PostingId original) {
  Posting source = postings.findById(original)...;
  if (source.isReversal()) { throw ReversalNotAllowedException.isItselfAReversal(original); }
  if (postings.existsReversalOf(original)) { throw ReversalNotAllowedException.alreadyReversed(original); }
  return write(Posting.reverseOf(PostingId.next(), source, clock.instant()));
}
```

The lock is taken later, inside `write` then `applyToBalances` (`Ledger.java:143-152`). The
losing thread therefore evaluates `existsReversalOf` on a stale read, blocks on the balance row,
and is let through only after the winner has committed ‚Äî but never re-evaluates the guard. It
then fails at whichever step comes first: the debit (`Balance.java:48`) or the `INSERT` against
`posting_reverses_at_most_once_idx`.

`V2__ledger.sql:48-54` shows the gap was understood but stopped one step short:

> The ledger refuses a second reversal before it gets here, but that check reads the table and
> then writes to it: two concurrent reversals can both read "not yet reversed". This index is
> what makes the second one fail.

The index does make it fail ‚Äî it just does not make it fail *as* `REVERSAL_NOT_ALLOWED`. BR-6
requires the rule to be "enforced in the ledger (not delegated to callers)"; surfacing an
untranslated `DataIntegrityViolationException` is precisely delegation to callers.

Because both reversals of one posting always contend on the same two balance rows, re-evaluating
`existsReversalOf` *after* the locks are held is sufficient and deterministic; translating the
constraint violation is the belt-and-braces half. Direction only ‚Äî the builder owns the fix and
the regression test.

### Transactions and locking

- `LedgerConfig.java:46-53` wraps `Ledger` in a `TransactionInterceptor` with
  `MatchAlwaysTransactionAttributeSource`, so every public method runs in a transaction. That is
  what makes `trialBalance()` read a consistent snapshot and what keeps `SELECT ... FOR UPDATE`
  held for the length of a posting.
- `applyToBalances` sorts the two ids and locks in ascending order (`Ledger.java:145-147`),
  matching BR-5. The debit is applied before the credit, so a refusal happens before anything is
  written.
- **Lock mutation re-verified independently.** The path-guard hook correctly refused a QA edit to
  `BalanceJpaRepository.java`, so the mutation was performed on an isolated copy of the backend
  outside the repository. With `@Lock(PESSIMISTIC_WRITE)` removed:

  | Test | Result without the lock |
  |---|---|
  | `LedgerConcurrencyIT.exactlyOneOfTwoConcurrentDrainsCommits` (builder) | fails ‚Äî `[COMMITTED, COMMITTED]` |
  | `LedgerConcurrencyIT.opposingTransfersDoNotDeadlock` (builder) | fails ‚Äî expected `BRL 85.0000` |
  | `LedgerRaceAcceptanceIT.exactlyOneOfTwoConcurrentDrainsCommits` (QA) | fails ‚Äî `expected: 1L but was: 2L` (x3) |
  | `LedgerRaceAcceptanceIT.opposingTransfersDoNotDeadlockOrLoseAnUpdate` (QA) | fails ‚Äî `BRL 60.0000` and `BRL 125.0000` where `BRL 85.0000` is correct (x3) |

  With the file untouched ‚Äî the repository was never modified, `git status` shows only QA-owned
  additions ‚Äî the same selection is green: `LedgerConcurrencyIT` 2/2 and the QA race suite 12/12.
  The builder's attestation is accurate: the race tests derive their power from the lock.

### Tests: do they carry real assertions

`LedgerSchemaIT` is the strongest of the builder's suites ‚Äî raw JDBC, going around the
application, and it asserts the value survived the refused `UPDATE` rather than only that an

### Architecture and conventions

- `ArchitectureTest.onlyTheLedgerTouchesBalances` is a real M5 rule and passes.
- No source file added or changed by this slice cites `CLAUDE.md`, a `SPEC-`, `OR-`, `BR-`, `DL-`

## Verdict (run 1 of 2): FAIL_REWORK

Money integrity held against every attack I could construct. I could not make a customer or box
account go negative by any application route, could not double-spend, could not get a posting
reversed twice, and could not mutate or delete a posting through the application or through raw
`UPDATE`/`DELETE`. The pessimistic lock is genuinely load-bearing, proven by independent
mutation. `trialBalance` really does audit the postings rather than agree with itself. `Money`
holds its four decimals and its half-up edge. That is a solid accounting core.

Rework is nonetheless required for two defects, both with executable repros and both cheap:

- **QA-01** breaks the error contract that acceptance criterion 7 and BR-6 state, and lets a
  Spring `dao` exception escape a `domain` API. Every money rail in the roadmap will consume this
  API; the cost of leaving it grows with each one.
- **QA-02** leaves a demonstrated bypass of M1, the append-only invariant, with a one-statement
  remedy already validated against the real schema.

QA-03 and QA-04 are recommended in the same cycle, both being small. QA-05 through QA-08 are
observations for the record and need no action to close this slice.

Per `.claude/rules/qa-ownership.md` the failing QA tests stay QA-owned: the builder writes the
regression tests from these repros and does not edit the acceptance suite.

```
=== ATTACK D: TRUNCATE posting, balance ===       ERROR: ... TRUNCATE on posting is not permitted
=== ATTACK F: DELETE / UPDATE ===                 ERROR: ... DELETE / UPDATE not permitted
=== SURVIVORS ===  postings = 2 | accounts = 5
```

Attack C is the one worth calling out: `TRUNCATE account CASCADE` reaches `posting` through the
foreign key, and PostgreSQL fires the cascaded table's own `BEFORE TRUNCATE` trigger, so the
indirect path is closed too and the whole statement rolls back. The ledger survives every variant.

`TRUNCATE balance CASCADE` still succeeds (attack E) ‚Äî see QA-07.

### QA-03 ‚Äî behaviour FIXED; its builder-owned regression test is vacuous (new finding QA-09)

### QA-09 ‚Äî the QA-03 regression test cannot fail

`LedgerMetricsIT.ignoresARolledBackPosting` provokes its rollback with an
`InsufficientFundsException` from `ledger.record`. On that path `applyToBalances` throws *before*
`postings.save` and therefore before `events.publish` is ever reached, so no `PostingRecorded`
event is published under either the old or the new listener. The test asserts that a counter which
was never going to move did not move.

Proven, not argued: with `LedgerMetrics` reverted to `@EventListener`, the class passes 3/3.

The bug it is supposed to pin is the one where a posting is recorded successfully and *then* the
surrounding transaction rolls back ‚Äî which needs an outer transaction, as
`LedgerMetricAcceptanceIT.aRolledBackPostingIsNotCounted` does. The behaviour is therefore not
unprotected: the QA-owned acceptance test is committed, runs in the battery, and fails against the
pre-fix code. But it is protected from the QA side, not where the commit message claims, and
`.claude/rules/qa-ownership.md` puts bug regression tests on the builder. If `LedgerMetricsIT` is
ever read as the guard for this behaviour, it will give false confidence.

### QA-10 ‚Äî the stable codes from QA-08 are asserted nowhere

`grep -rn "UnknownAccountException\|UnknownPostingException\|UNKNOWN_ACCOUNT\|UNKNOWN_POSTING"
backend/src/test/` returns nothing. The QA acceptance suite's `refusesToReverseAnUnknownPosting`
asserts only `RuntimeException`, deliberately, because when it was written the production code
still threw `NoSuchElementException` and the loose assertion was what kept the observation from
becoming a false failure.

The whole point of QA-08 was to give these failures a stable code that callers can react to. A
revert to `NoSuchElementException` would be caught by nothing today. Low severity ‚Äî the fix is
correct and the surface is small ‚Äî but on an R3 slice whose value proposition is a stable error
contract, an untested contract is worth the owner seeing.
