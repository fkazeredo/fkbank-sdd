# FKBANK 0.1.0 — internal pilot

**Release class:** internal pilot / pre-release. Not a production, end-user-facing release
(`.claude/workflow-policy.yml` §production_release_definition).

**Security assurance:** `SECURITY_OBSERVATIONS` — `docs/security/reports/SPRINT-1-3dec822.md`.
Every applicable mandatory control executed. `SECURITY_VERIFIED` is not claimed: the
repository-configuration control fails because no branch protection is configured, a risk the
owner accepted for this window (`SEC-0001` in `docs/security/DECISIONS.md`). Risk acceptance
never produces `SECURITY_VERIFIED`.

## What this release is

Sprint 1's outcome: a walking skeleton observable from day one, the accounting heart proven by
race tests, and a person opening an account and signing in.

- **SPEC-0018 — walking skeleton.** Three-root architecture with ArchUnit and Modulith gates,
  default-deny authorization behind an embedded Authorization Server (OIDC + PKCE), Flyway over
  PostgreSQL, an Angular shell behind a single published edge, an end-to-end stack, and CI.
- **SPEC-0016 — observability baseline.** Per-request correlation id propagated into async work
  and every structured JSON log line, an authorization-failures counter on a token-protected
  Prometheus endpoint, and a public drift-checked OpenAPI document.
- **SPEC-0001 — ledger core.** Append-only double-entry postings, four-decimal `Money` rounded
  at the edge only, balances that cannot go below zero, reversal-at-most-once by contra-posting,
  and a trial balance that audits the stored balances against the postings — proven serialized
  under `FOR UPDATE`.
- **SPEC-0002 — sign-up and account opening.** Sign-up, automated KYC against the bureau
  emulator, account opened with its opening `$0.00` posting, OIDC + PKCE sign-in, and a zero
  balance on the new account. Same-CPF sign-ups converge to exactly one customer under real
  persistence; sign-up is idempotent by CPF while a check is pending.

## Fixed after the merge, before this release

- The mandatory R3 review of SPEC-0002 had not run before its merge; it was run during closure
  and found a High: a rejected sign-up rendered a general "review the details" with no field
  highlighted, and a valid-looking full name could dead-end at the server. The frontend now
  reads the problem+json a 422 actually returns and mirrors the server's full-name rule inline.
- The security assurance track's dynamic (DAST) family had never executed on any candidate — a
  `curl`-based container healthcheck on a `curl`-less image. Fixed; the DAST scan now runs, and
  the edge gained the security headers (CSP, anti-clickjacking, `nosniff`, `Permissions-Policy`)
  it had been shipping without.

## Known limitations carried into Sprint 2

Recorded with severity and a concrete failure scenario in `docs/qa/SPRINT-1-CLOSURE.md` §7.
None blocks an internal pilot; the following are required before any production release:

- Raw CPF can reach application logs through a database driver's error message on the
  duplicate-CPF path, in the production profile (`SEC-F-01`, CWE-532).
- The leap-day coming-of-age rule is enforced on the server but not in the browser, so a
  29-February birth date can block the form on the eve of a birthday.
- E-mail uniqueness is unenforced while an application is pending, so two pending applications
  can share an e-mail and strand the second at approval.
- A pending application has no automatic recovery or expiry path.
- Branch protection is not configured on the repository (`SEC-0001`, owner-accepted).

## Notes for the next release

- The bureau emulator (`emulators/bureau`) is a test double of an external system and is not a
  shipped artifact; it remains on `-SNAPSHOT` while the application modules carry `0.1.0`.
- `tools/release/set-version.ps1` fails on this Windows host because `Start-Process` returns a
  null exit code for a `.cmd` wrapper; the version files here were set with the same underlying
  `mvnw versions:set` and `npm version` the tool invokes. Worth fixing before the next release.
