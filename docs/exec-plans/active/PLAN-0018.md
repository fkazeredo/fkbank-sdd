# PLAN-0018 — Walking skeleton

- **Branch:** `feature/spec-0018-walking-skeleton`

## Understanding

Raise the thinnest end-to-end thread of the documented architecture and make every gate that
guards it executable from the first build: a person logs in through OIDC/PKCE against the
embedded Authorization Server, and the authenticated Angular shell calls one protected
endpoint (`GET /api/me`). Nothing in this slice is business behavior — its product is the
*baseline* (structure, security posture, gates, stack) that SPEC-0016, SPEC-0001 and
SPEC-0002 all assume already exists.

The single most important consequence: the architecture rules stop being prose and become
build failures. ArchUnit + `ApplicationModules.verify()` + JaCoCo floors are bound to
`./mvnw verify`, so a later slice cannot silently drift out of the three-root hexagonal
structure.

## Behavioral domain modeling (CLAUDE.md invariant 9)

`domain.identity` is the first domain code in the repository, so it sets the precedent.

- `Username` — value object, **record** with a compact constructor that trims, lowercases and
  rejects blank/oversized input. Permitted as a record by ARCHITECTURE §Behavioral domain
  modeling: it cannot represent invalid state and owns its normalization behavior.
- `AuthenticatedUser` — **regular class**, not a record: it is the lifecycle-bearing identity
  concept that later slices grow (transaction PIN, lockout, authorization policy — DOMAIN
  §Module map). Private field, no setter, no all-args constructor, static ubiquitous-language
  factory `signedIn(Username)`, defensive null rejection.
- `MeResponse` — **record**, in `application.api`. Boundary DTO; the domain entity never
  becomes JSON (invariant 6).

**Deliberately not modeled:** a `Permission`/policy aggregate. Default-deny in this slice is
enforced by `infra.security` configuration, and no acceptance criterion asks the domain to
decide a permission. Inventing a policy model now would be speculative (Rule 0 / rung 1);
identity's authorization behavior arrives with SPEC-0006 and SPEC-0013. `AuthenticatedUser`
is therefore *thin by scope*, not anemic by neglect — it still owns valid construction and
exposes ubiquitous language rather than raw state assembly.

## Decision Ladder

1. **Necessity / YAGNI.** Every acceptance criterion maps to a deliverable; nothing beyond
   them is planned. Three Scope items are already served by the baseline (table above) and are
   satisfied by evidence rather than by new code. Out-of-scope items named by the spec
   (actuator, metrics, structured logs, springdoc, business tables, emulators, PIT, VM
   cutover) are not started.
2. **Existing project reuse.** The whole baseline table above is reused as-is — no new compose
   file, no new CI workflow, no second preflight, no reimplemented verification battery. The
   only additions are the two Dockerfiles `compose.e2e.yaml`/`compose.prod.yaml` already
   reference but that do not exist yet.
3. **Standard library / primary framework.** Spring Boot starters (`web`, `security`,
   `oauth2-resource-server`, `oauth2-authorization-server`, `data-jpa`, `validation`), Flyway,
   Spring Modulith, JUnit 5, Testcontainers, ArchUnit, JaCoCo — all named in ARCHITECTURE
   §Stack, none new. Frontend: Angular 22 standalone/zoneless/signals, `HttpClient`,
   functional interceptor and router guard, `crypto.subtle`, `URL`/`URLSearchParams`.
4. **Browser / platform capability.** The OIDC + PKCE authorization-code flow is implemented
   on browser primitives: `crypto.getRandomValues` for the verifier, `crypto.subtle.digest`
   for the S256 challenge, a redirect to the embedded Authorization Server and a token
   exchange with `fetch`/`HttpClient`. Access token held in memory only (no `localStorage`) —
   nothing to steal from persistent storage in a banking SPA.
5. **Already-installed dependency.** No `pom.xml`/`package.json` exists yet, so this rung is
   vacuous — every dependency is chosen at rung 3 from the architecture-approved stack.
6. **Simplest clear solution.** `application.api` translates transport into a domain call
   directly (ARCHITECTURE §Style: "application translates transport input into calls to domain
   use cases"). No argument resolver, no `*Impl`, no use-case/interactor class, no
   orchestration service for a read that has no orchestration.
7. **Minimum new code.** Only the approved thread. Tests proportional to R3: the security and
   architecture surfaces get real assertions, the thin identity model gets unit coverage.
8. **New external dependency = last resort.** **None added.** Explicitly rejected:
   `angular-oauth2-oidc`. Rungs 3–4 already answer the need — the SPA drives one flow against
   *our own* Authorization Server, needing no discovery, silent renew or multi-provider
   support, and the real security boundary is server-side token validation by Spring's
   resource server, not the client redirect choreography. Adopting a library here would add a
   supply-chain surface for roughly a hundred lines of standard, testable browser code.
   *Flagged for QA and `/review-pr` focus, since it is the security-sensitive judgement call
   of this slice.*

## QA focus

Adversarial on the authorization baseline: probe for a route reachable without a token; assert
the `401` body carries no stack trace or internal detail; confirm the seeded credential is
absent outside `dev`/`e2e`; confirm `prod` aborts **before opening a port** on a dev default;
verify the E2E journey is green on the *first* run (no retry masking). Confirm the CI jobs
executed the real build rather than the pre-bootstrap no-op.

## Risks

- **Testcontainers needs a Docker daemon** — absence fails the build loudly; never skipped.
- **Angular 22 + zoneless + PrimeNG** — asserted by Vitest running zoneless from day one.
- **Port collision on the dev box (DL-0005)** — mitigated by the single IPv4 origin
  `127.0.0.1:8090` and the existing preflight; never `localhost` (IPv6 first on Windows).
- **Hand-rolled PKCE** — the deliberate rung-8 rejection above; mitigated by integration-level
  proof of the real token exchange and flagged for independent review.

## One-session fit check

**Verdict: fits as one slice — build it whole.**

The spec's `split_review_required: true` demands this evaluation, and the split boundary it
names is backend+CI vs frontend+E2E. Against the slice-too-big signals in
`.claude/rules/decision-ladder.md`:

- *More than three independent vertical behaviors?* No — there is exactly **one** vertical
  behavior (log in → read the authenticated principal). The volume is structural setup around
  a single thread, not many behaviors.
- *Parts releasable/testable separately?* No, and this is decisive. The spec requires the
  walking-skeleton thread to land whole; a backend-only slice cannot demonstrate the PKCE
  journey, and a frontend-only slice has nothing to authenticate against. Splitting would
  produce two slices neither of which satisfies a single acceptance criterion end to end.
- *Structural change + feature together?* Yes, unavoidably — that *is* the deliverable of a
  walking skeleton, not an accidental coupling.
- *Expected compaction during `/build`?* Bounded by sequencing the work as the three commits
  above, each independently verifiable.

Split is therefore rejected as harmful to the slice's purpose. Contingency: if compaction does
occur mid-build, the resumable boundary is exactly the commit-2/commit-3 seam (backend proven,
frontend pending), which is the same seam the spec names — so the fallback stays available
without planning for it up front.
