# PLAN-0002 — Sign-up and account opening

Spec: `docs/specs/SPEC-0002-signup-account.md` · Risk: R3 · Approved hash:
`3245eef9fc8e761ead5279aca5570146e78fdd17279ef2cc42c0a36991bfceb7`

## Understanding

A person who is not yet a customer becomes one, alone, and lands on a home screen showing an
account with no money in it. Five things have to be true at the end: the ledger gained a
customer account, the identity module gained a credential that can actually log in, the same
CPF can never produce two customers, a rejected applicant gets a reason without seeing the
bureau's raw answer, and an applicant whose bureau call timed out is not stranded.

This is the first slice with an external system, so it is also the first with an
anti-corruption layer, a signed callback and an emulator service. It is the first slice with a
public route, the first with a form, and the first that issues a real credential — until now
the product logged in as a seeded developer user.

Four bounded contexts, in dependency order: `customer` and `identity` depend on nothing,
`account` depends on `ledger`, and `onboarding` orchestrates the three.

## Decision Ladder

Run once, in full, as the rules require.

**1 — Necessity.** Every acceptance criterion maps to behaviour below and nothing is built
that no criterion or explicitly named requirement asks for. Two requirements were checked
against the ladder's "an explicit requirement is never silently dropped" rule and escalated
rather than resolved here: the `duplicate-webhook` scenario (DL-0006) and the emulator's
technology (DL-0007). Both were decided by the owner before this plan was written.

**2 — Existing project reuse.** Reused as-is: `Ledger.openAccount` and the chart-of-accounts
kinds for the `$0.00` opening; the Modulith event-publication registry for `OnboardingApproved`
and `AccountOpened`; `ProblemDetailAuthenticationEntryPoint`'s problem-response shape; the
`LedgerConfig` transaction-interceptor pattern; `PostgresContainer` and the shared integration
base class; the PKCE flow, auth interceptor and guard in the frontend; the `messages.ts` i18n
table. Deliberately **not** reused: the ledger's `Money` for declared monthly income — the
`customer` context may not depend on `domain.ledger`, and self-reported income is not a
balance (DL-0004).

**3 — Standard library / framework.** `java.time` for the age rule; Bean Validation for edge
shape checks; `javax.crypto.Mac` (`HmacSHA256`) for the callback signature — no library;
Spring Security's `DelegatingPasswordEncoder` for hashing; Spring Data for repositories;
`RestClient` with explicit timeouts for the bureau; Angular reactive forms and `Intl` through a
money pipe.

**4 — Platform capability.** A partial unique index enforces "one pending onboarding per CPF"
in the database rather than in application code, which is the only place it survives a race.
Postgres sequences generate the account number. `Intl.NumberFormat` formats money.

**5 — Already-installed dependency.** No new runtime dependency in the backend. The emulator
module uses the Spring Boot starters already in the reactor.

**6 — Simplest clear solution.** Two domain classes carry the sign-up flow instead of one:
`SignUp` orchestrates and holds no transaction, `OnboardingOutcome` applies a bureau decision
inside one. The split is not decoration — it keeps the external HTTP call outside a database
transaction, and it gives the webhook a single entry point to reuse, which is what makes
duplicate delivery a no-op rather than a special case.

**7 — Minimum new code.** No generic `Idempotency-Key` filter is built: this slice exposes no
money-moving route, and its idempotency is the natural-key form (CPF while pending, onboarding
id on the callback). The header mechanic arrives with the first money route that needs it.

**8 — New external dependency.** None.

### One-session fit check

Six of eight slice-too-big signals fire: four new bounded contexts, a structural change (the
first emulator service) alongside a feature, a new cross-context contract plus full UI, an
external integration, a migration with behaviour, and likely compaction during `/build`. The
natural seam is `OnboardingApproved → AccountOpened`. **The owner decided to build it whole**
(DL-0005): halves of this journey ship no customer value.

## Design

### Flow

```
POST /api/signup ──► SignUp.submit
                       ├─ reject invalid input                         422
                       ├─ customer with this CPF or e-mail exists?     409 DUPLICATE_CUSTOMER
                       ├─ pending onboarding with this CPF?            200 + that onboarding
                       ├─ start onboarding PENDING                     (unique index decides races)
                       └─ ask the bureau  ──► APPROVED / REJECTED ──► OnboardingOutcome.apply
                                          └─ timeout ─────────────► stays PENDING, 202

POST /api/webhooks/bureau ──► verify HMAC ──► OnboardingOutcome.apply
                                └─ bad signature                       401, nothing written
                                └─ onboarding no longer PENDING        200, nothing written
```

`OnboardingOutcome.apply` is one transaction: it approves or rejects the onboarding, and on
approval creates the customer, activates the credential, opens the account, opens the
customer's ledger account with a `$0.00` materialized balance, and publishes
`OnboardingApproved` and `AccountOpened`. Per DL-0002 there is **no zero-amount posting row** —
the ledger forbids one.

### Domain

`domain.customer` — `Customer` (class; `register(...)` enforces the 18+ rule), `CustomerId`,
`Cpf` (record; normalizes formatting to digits, validates check digits), `Email`, `FullName`,
`BirthDate`, `MonthlyIncome` (record; non-negative, 2 decimals), `CustomerRepository`,
`CustomerException` + `DuplicateCustomerException` (`DUPLICATE_CUSTOMER`).

`domain.identity` — adds `Credential` (class; `issue(...)`, `activate()`, `isActive()`),
`CredentialId`, `RawPassword` (record; enforces the eight-character letter-and-digit rule,
masked `toString`, never persisted), `PasswordHash`, `PasswordHasher` (port),
`CredentialRepository`.

`domain.account` — `Account` (class; `openFor(customerId, number)`), `AccountNumber` (record;
branch `0001` plus a sequential number), `AccountOpened` (event), `AccountRepository`,
`AccountNumbers` (port over the sequence). Depends on `ledger` only to call
`Ledger.openAccount`; it stores no ledger identifier, deriving the account code from the
customer id.

`domain.onboarding` — `Onboarding` (aggregate; `approve()`, `reject(reason)`, `isPending()`),
`OnboardingId`, `OnboardingStatus`, `RejectionReason` (the category shown to the person),
`SignUp`, `OnboardingOutcome`, `BureauCheck` (port), `BureauDecision`, `OnboardingApproved`,
`OnboardingRepository`.

### Persistence — `V3__signup_account.sql`

`customer` (unique `cpf`, unique `email`), `credential` (unique `username`, unique
`customer_id`), `account` (unique `customer_id`, unique `(branch, number)`, backed by a
sequence), `onboarding` (status, reason category, nullable `customer_id`) with a **partial
unique index on `cpf` where `status = 'PENDING'`** — this is what makes the concurrent
same-CPF race resolve to exactly one row, and it still lets a rejected applicant try again.

### Security

The sign-up route, the status route and the callback are public and must be added to
`SecurityConfig`'s allowlist and to `RoutePermissionCompletenessIT`'s. The status route is
readable only with the opaque onboarding id and returns a status and a reason category, never
registration data. The callback authenticates by HMAC over the raw body; a bad signature is a
`401` before anything is read. `PasswordEncoder` moves out of the dev-only `SeededLoginConfig`
into an unprofiled configuration, and `UserDetailsService` reads the `credential` table, which
retires the seeded developer user that SPEC-0018 left as a placeholder. The bureau's raw
payload never leaves the anti-corruption layer, and CPF is masked in logs.

### Emulator — `emulators/bureau`

Business API `POST /inquiries`; control API `POST /control/scenario` selecting `approve`,
`decline`, `delay` or `duplicate-webhook`, globally or per CPF. `delay` answers past the
client's timeout and then calls back; `duplicate-webhook` delivers the same callback twice.
In-memory state, deterministic under a seed, container in the `emulators` compose profile and
in the E2E stack.

### Frontend

`features/signup/` (the first reactive form: full name, CPF, e-mail, password, birth date,
declared monthly income, with field-level errors), `features/login/` (a public landing that
starts the OIDC redirect and links to sign-up — the SPA never sees the password),
`features/account/` (home: balance and account details). `shared/ui/money.pipe.ts` is the
single money formatter the architecture requires. New public routes sit outside the guarded
shell. Every string goes through `messages.ts`.

## Risks

The seeded-login removal touches every existing authenticated test, and `PkceTokenFlow`
hardcodes the seeded credentials — it becomes parameterized and its callers create a real
customer first. `ProductionSecretsGuard` and `AuthenticationProperties` lose part of their
reason to exist and must be reduced without weakening the issuer check. Adding a container to
the E2E stack lengthens it. The `account` context introduces a second type named `AccountId`
in the codebase; the account aggregate deliberately holds no ledger identifier so the two never
meet in one file.
