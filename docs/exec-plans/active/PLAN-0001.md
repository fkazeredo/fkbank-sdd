# PLAN-0001 — Ledger core

Spec: `docs/specs/SPEC-0001-ledger-core.md` · Risk: R3 · Approved hash:
`cd29e975d28c191d6e593a6e4d8aae9705ee89e22db252010af2f8ab653a61e4`

## Understanding

The accounting heart. This slice creates the only code in the product allowed to touch a
balance, and the schema that makes tampering fail loudly. Every later money rail
(boleto, PIX, transfer, card, box, loan) commands this module and adds no balance logic of
its own.

Three behaviours, one aggregate, no delivery surface:

1. **Record** a double-entry posting — atomic, immutable, never driving a customer account
   below zero.
2. **Reverse** a posting exactly once, by writing a contra-posting, never by mutating history.
3. **Verify** the books — every materialized balance equals the sum of its own postings, and
   total debits equal total credits.

There is no REST endpoint, no screen and no rail in this slice. The callers are tests.

## Modules touched

- `domain.ledger` — new bounded context (flat, Modulith-annotated). All banking behaviour.
- `infra.persistence.ledger` — JPA entities, Spring Data repositories, port implementations.
- `infra.configuration` — the transactional decorator around the ledger and its bean wiring.
- `infra.observability` — the `fkbank.ledger.postings` counter, fed by the domain event.
- Migration `V2__ledger.sql` — chart of accounts, postings, balances, append-only trigger,
  single-reversal index, Modulith event-publication table.

## Patterns to reuse

- `domain.identity.Username` — the validating-record shape, `of(...)` factory, self-contained Javadoc.
- `domain.identity.package-info` — the `@ApplicationModule` annotation and the reason it is the
  single permitted Spring reference under `domain`.
- `testsupport.PostgresContainer` — the shared PostgreSQL 16 container; no new container plumbing.
- `FlywayBaselineIT` — the JDBC-level migration assertion style. **This test asserts exactly one
  migration and zero business tables; V2 invalidates both assertions by design and the builder
  updates it in the same commit.**
- `ArchitectureTest` — extended with the balance-confinement rule, not rewritten.

## Contracts / data / migrations

- **No public contract.** No REST route, no DTO, no OpenAPI change, no frontend file.
- **Cross-context contract:** `Money` and the ledger ports become the vocabulary later contexts
  depend on. `Money` lives in `domain.ledger` because ArchUnit forbids any type sitting directly
  in `com.fkbank.domain`; the ledger is its rightful owner anyway.
- **Migration is additive**, and the first business schema in the product. `V1__baseline.sql`
  stays untouched — an applied migration is immutable.

## Decision Ladder

1. **Necessity / YAGNI.** Every acceptance criterion maps to behaviour that must exist:
   posting, the never-below-zero rule, reversal, the ordered pessimistic lock, verification,
   `Money` rounding, single-reversal enforcement. Nothing speculative is planned. Deliberately
   **not** built: idempotency (the spec excludes it — it lands at the REST edge with the first
   money route), multi-currency behaviour (field present, BRL only), statement projections
   (SPEC-0003), any REST surface. `openAccount` is included only because criteria 2 and 4 are
   untestable without a customer account.
2. **Existing project reuse.** `PostgresContainer`, the `Username` record shape, the
   `package-info` module pattern, the existing `ArchitectureTest` (extended), the existing
   Flyway pipeline. No new test infrastructure.
3. **Standard library / framework.** `BigDecimal` + `RoundingMode.HALF_UP` for money,
   `java.util.Currency` for the currency field, `java.time.Instant` for UTC timestamps,
   Spring Data JPA `@Lock(PESSIMISTIC_WRITE)` for `SELECT ... FOR UPDATE`, Spring Modulith's
   JPA event registry as the outbox base. No hand-rolled money type beyond the value object,
   no custom locking, no custom outbox.
4. **Browser / platform capability.** The database enforces what the database can enforce:
   append-only by trigger, single reversal by partial unique index, atomicity by transaction.
   These are not re-implemented in Java as the primary guard — the domain raises the business
   error, the schema makes the corrupt state impossible.
5. **Already-installed dependency.** JPA, Flyway, PostgreSQL driver, Testcontainers, Modulith
   core, ArchUnit, Micrometer, Lombok are already in `backend/pom.xml`.
6. **Simplest clear solution.** One aggregate, three ports, one domain class as the internal
   API. No CQRS, no event sourcing, no repository abstraction beyond the three ports actually
   used, no generic "domain service" layer.
7. **Minimum new code.** Only the approved behaviour. The transactional decorator is the
   mechanism the architecture already prescribes, not a new invention.
8. **New external dependency.** Two, both mandated by already-approved documents rather than
   chosen here:
   - **jqwik** — `docs/ARCHITECTURE.md` §Testing: "jqwik mandatory on the math"; and the spec's
     sixth acceptance criterion names it verbatim. Test scope only.
   - **spring-modulith-starter-jpa** — the spec's DL-0004 fixes the JPA event registry as the
     outbox base, and the roadmap's follow-through note has the outbox gauge sampling
     "the event-publication registry SPEC-0001 introduces".
   - **PIT (pitest)** — `docs/ARCHITECTURE.md` §Testing sets a ≥60% floor on money-moving
     modules, of which the ledger is the first. Configured behind a profile so it stays a
     QA-invoked check rather than a cost on every build.
   Record: `docs/adr/ADR-0001-ledger-dependencies.md`.

## TDD sequence (behaviour-sized steps)

1. `Money`: construction, scale normalization, `add`/`subtract`/`negate`, `atEdge()` half-up
   rounding, `split` remainder-on-last. Unit + jqwik properties. **No database yet.**
2. `AccountKind` / `Account`: which kinds tolerate a negative balance.
3. `Balance`: `debit` past zero raises `InsufficientFundsException` for customer accounts and
   succeeds for internal accounts. Pure unit.
4. `Posting`: rejects zero/negative amounts and an account posting against itself; `reverseOf`
   swaps the accounts and references the original.
5. Migration `V2` + JPA mappings: schema applies; the append-only trigger rejects `UPDATE` and
   `DELETE`; the partial unique index rejects a second contra for one original. Integration.
6. `Ledger.record` against PostgreSQL: balances move, each equals the sum of its own postings,
   Σ debits = Σ credits (criterion 1).
7. `Ledger.record` refusing to overdraw: nothing written, every balance unchanged (criterion 2).
8. `Ledger.reverse`: contra written, balances restored, original row untouched (criterion 3).
9. `Ledger.reverse` twice, and reversing a reversal: `REVERSAL_NOT_ALLOWED`, nothing written,
   balance keeps its single-reversal value, verification still consistent (criterion 7).
10. Race: two concurrent drains of a single-funded account — exactly one commits, the other
    raises `INSUFFICIENT_FUNDS`, final balance ≥ 0, Σ debits = Σ credits (criterion 4).
11. `Ledger.trialBalance`: flags a tampered account, flags none on a clean ledger (criterion 5).
12. ArchUnit: no balance access outside `domain.ledger` (M5).

## Builder test plan

- **Unit (pure domain, no Spring):** `MoneyTest`, `MoneyPropertiesTest` (jqwik), `BalanceTest`,
  `PostingTest`, `AccountKindTest`, `TrialBalanceTest`.
- **Integration (Testcontainers PostgreSQL 16):** `LedgerPostingIT`, `LedgerReversalIT`,
  `LedgerVerificationIT`, `LedgerConcurrencyIT` (the mandatory race test),
  `LedgerSchemaIT` (trigger + partial index), updated `FlywayBaselineIT`.
- **Architecture:** `ArchitectureTest` extended with balance confinement; `ModulithTest`
  re-verifies with the new module present.
- The race test is deterministic: a `CountDownLatch` releases both threads together, both
  target the same pair of accounts, and the assertion is on outcomes (one commit, one
  `INSUFFICIENT_FUNDS`) plus the domain invariants at the end — never on timing.

## QA focus

- Execute every acceptance criterion literally against real PostgreSQL, not an in-memory profile
  (criterion 4 says so explicitly).
- Adversarial: try to corrupt the ledger. `UPDATE`/`DELETE` a posting directly; insert a second
  contra by hand; drive an internal account negative (must be allowed) and a customer account
  negative (must not); check that `trialBalance` actually detects tampering rather than
  recomputing its way to agreement.
- Verify the race test fails when the lock ordering or the `FOR UPDATE` is removed — a race test
  that passes without the lock is proving nothing.
- Confirm `Money` never leaks 4-decimal values where 2 are expected, and that no rounding happens
  mid-calculation.

## Risks

- **The race test passing for the wrong reason.** Mitigated by a deliberate mutation: remove the
  lock, watch it fail. QA re-checks this independently.
- **`FlywayBaselineIT` breaking loudly.** Expected and planned, not a surprise — its assertions
  are about the walking skeleton's emptiness, which this slice ends.
- **BigDecimal equality.** Scale is normalized to 4 in the compact constructor so record equality
  and database round-trips agree. Covered by a property test.
- **Modulith event registry schema.** Created explicitly in `V2` rather than auto-generated, so
  the schema stays under Flyway's control.
- **The transactional decorator.** Class-proxying the domain `Ledger` is the mechanism the
  architecture prescribes; if it silently fails to apply, the race test is the thing that catches
  it (no transaction ⇒ no lock ⇒ both drains commit).

## Non-goals

No REST endpoint · no idempotency mechanism · no statement or receipt · no rail · no screen ·
no multi-currency conversion · no scheduled verification job (the routine is callable; wiring it
to a scheduler belongs to the slice that needs it) · no repository-wide refactor of existing code.

## Open doubts (batched)

None material. Two written-rule resolutions applied rather than asked:

- `docs/ARCHITECTURE.md` §Behavioral domain modeling names `domain.<context>.port` for repository
  contracts, while the same document, CLAUDE.md invariant 6 and an executable ArchUnit rule all
  require bounded contexts to be flat. The executable rule and the existing `domain.identity`
  layout decide: ports sit flat in `domain.ledger`. Recorded here because the prose is stale.
- The three dependencies in rung 8 are mandated by the approved spec and architecture, so they are
  derived rather than chosen. Their records are filled for durability.

## One-session fit check

**Fits — with the density flagged.**

Against the slice-too-big signals: one bounded context · one migration · no UI · no REST · no
cross-context contract · no external integration · no backfill · no independent rollout. The
three behaviours (record, reverse, verify) are not independent verticals — they are one
aggregate's lifecycle, and splitting them would produce a ledger that can post but not
reverse, which is not releasable.

The honest counterweight: seven acceptance criteria, a Testcontainers race test, a jqwik
property suite and the product's first business migration make this the densest slice of
Sprint 1 after the skeleton. The mitigation is the ordered TDD sequence above — steps 1–4 are
pure domain and carry no infrastructure cost, so the expensive integration work starts against
an already-proven model.
